# Scraper Plugin Upload UI — Milestone 6: `.zip` Accept + `bankId` Success Message

**Date:** 2026-03-21
**Scope:** Frontend-only patch — no backend changes, no Prisma migration, no new API endpoints.
**Goal:** Align `PluginManager.tsx` with the updated `POST /admin/scrapers/install` contract:
- File input must accept `.zip` (not `.js`)
- Success feedback must reference `bankId` (not `filename`)
- The generated OpenAPI model (`installPluginResponseDto.ts`) must reflect the new DTO shape
- The unit test file must be updated to match

---

## 1. Background — Contract Divergence

### Backend DTO (current, confirmed)

File: `packages/backend/src/scraper/install-plugin-response.dto.ts`

```
message: string   — Human-readable summary
bankId:  string   — bankId from the plugin's default export
pluginDir: string — Absolute path of the directory written to SCRAPER_PLUGIN_DIR
```

The `filename` field **no longer exists** on the backend DTO.

### OpenAPI snapshot (stale)

File: `packages/frontend/openapi.json` — `InstallPluginResponseDto` schema at line 3538 still declares:
```json
"properties": { "message": {...}, "filename": {...} },
"required": ["message", "filename"]
```
This snapshot is **out of date**. The generator would produce the wrong types if run from the snapshot.

The `file` field description in the request body (line 1898) still reads `"A .js file whose default export satisfies BankScraper"` — it too is stale but does not affect runtime behaviour (it is only a doc string on `ScraperAdminControllerInstallBody.file: Blob`).

### Generated model (stale)

File: `packages/frontend/src/api/model/installPluginResponseDto.ts` — generated by orval v8.4.2 from the stale snapshot. Contains `filename: string` instead of `bankId: string` and `pluginDir: string`.

## 2. Files to Change

### 2.1 `packages/frontend/openapi.json` — update snapshot before regenerating

This file is the orval input. It must be updated to reflect the new backend contract **before** running `npm run generate:api`, otherwise the generator will re-emit the old types.

The orval config (`packages/frontend/orval.config.ts` line 24) targets `./openapi.json`. Running `npm run generate:api:live` fetches from the live backend and regenerates, but the standard `npm run generate:api` reads the file directly.

**Two options (choose one):**

- **Option A — Live regeneration (preferred when backend is running):** Run `npm run generate:api:live` in `packages/frontend`. This calls `openapi:fetch` (which writes a fresh `openapi.json`) and then `generate:api`. This automatically fixes `openapi.json` and all generated files in one step.

- **Option B — Manual snapshot edit + generate:** Manually edit `openapi.json` to update the `InstallPluginResponseDto` schema, then run `npm run generate:api`. The exact changes to `openapi.json`:
  - Remove the `"filename"` property block (lines 3546–3549).
  - Add two new property blocks: `"bankId"` (`string`, description `"bankId from the plugin's default export"`, example `"cibc"`) and `"pluginDir"` (`string`, description `"Absolute path of the directory written to SCRAPER_PLUGIN_DIR"`, example `"/plugins/cibc"`).
  - Update `"required"` array from `["message", "filename"]` to `["message", "bankId", "pluginDir"]`.
  - Update the `file` field description in the `POST /admin/scrapers/install` request body from `"A .js file whose default export satisfies BankScraper"` to `"A .zip package whose default export satisfies BankScraper"`.

Regardless of the option chosen, the generated files below will need to be in sync with each other before the component edit.

## 6. Frontend Test Scope (for frontend-tester)

This is a patch to a single component — no new screens, routes, or pages. Coverage level: **regression**.

### User flows to cover

1. **Happy path — successful install with `message`:** Select a `.zip` file, click Install, mutation resolves with `{message: 'Plugin installed', bankId: 'cibc', pluginDir: '/plugins/cibc'}`. Assert success banner shows `'Plugin installed'`.

2. **Happy path — install fallback (empty `message`):** Mutation resolves with `{message: '', bankId: 'cibc', pluginDir: '/plugins/cibc'}`. Assert success banner shows text containing `cibc`.

3. **No file selected:** Click Install without selecting a file. Assert error banner shows `'Please select a .zip plugin package'`.

4. **File input accept filter:** The rendered `<input type="file">` must have `accept=".zip"`.

5. **Aria label updated:** `getByLabelText(/select a .zip plugin package/i)` resolves to the file input.

6. **Error path — install fails:** Mutation rejects. Assert error banner shows `'Failed to install plugin'`.

### Preconditions

- Auth: user must be logged in with `ADMIN` role (or component is rendered in isolation with mocked hooks).
- For unit tests: mutation hooks are mocked via `vi.mock('@/api/admin/admin.js')`.
- For E2E (if run): backend running at `localhost:3001`, admin user seeded, a valid `.zip` plugin package available on the tester's filesystem.

### Notes

- The `.js` accept filter, `data.filename` reference, and "Please select a .js plugin file" guard message are the three regression targets. Any test that passes with the old strings after the patch is a false positive.
- There is no new screen or route — the `/admin` page and `AdminPage.tsx` composition are unchanged.
